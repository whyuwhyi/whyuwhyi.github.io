# Ventus 专栏执行规范（中文）

## 1. 目标与范围

本仓库维护一个长期更新的 `Ventus GPGPU` 学习专栏，目标是：

1. 形成一套可持续扩展的系统化解读文档；
2. 覆盖“总览 + 各子系统深挖 + 仿真与验证”；
3. 每篇内容都能落到真实代码路径与可复现实验信息；
4. 保持教程可读性，同时保留个人学习记录价值。

专栏不限制篇数，按学习进度持续扩展。

## 2. 写作语言与风格

1. 所有文档均使用中文编写。
2. 文章主风格为教程型：先讲问题和结构，再讲代码与验证。
3. 保留学习笔记属性：明确写出“当前结论、未决问题、下一步计划”。

## 3. 专栏组织方式

1. 专栏入口页：`source/ventus/index.md`。
2. 文章统一放在：`source/_posts/`。
3. 分类规则：
- 一级分类固定：`Ventus 专栏`
- 二级分类按系统划分，如：`总览`、`CTA 调度`、`Pipeline`、`内存系统`、`仿真与测试`。
4. 标签至少包含：`Ventus` + 当前模块标签。

## 4. 每篇文章最小结构（必须满足）

1. 背景与问题定义：本篇要解决的理解问题。
2. 模块职责与边界：输入/输出/上下游关系。
3. 数据流或控制流图：优先使用 Mermaid。
4. 关键代码解读：片段 + 文件路径 + 行为解释。
5. 场景走读：至少一个“输入到输出”的完整过程。
6. 验证与观察：日志、仿真命令、或可核对现象。
7. 小结与遗留问题：下一篇承接点。

## 5. 图示与代码引用规范

1. 图示优先 Mermaid，必要时补充仓库已有图片（如 `ventus-env/gpgpu/docs/images`）。
2. 代码引用采用“关键路径片段”原则，避免大段粘贴。
3. 每段代码必须标注来源文件路径（例如：`ventus/src/cta/cta_scheduler.scala`）。
4. 禁止“只有代码没有解释”。

## 6. Front Matter 规范

每篇专栏文章必须包含：

- `title`
- `date`
- `tags`
- `categories`
- `description`
- `toc: true`

推荐额外字段：

- `mathjax`（按需）
- `mermaid: true`（如主题解析需要时启用）

## 7. 验收标准（发布前检查）

1. 内容结构完整：满足“最小结构”7节。
2. 至少 1 张 Mermaid 图。
3. 至少 2 段关键代码片段且含路径。
4. 至少 1 个场景走读或状态机流程。
5. 文章中所有路径和术语可在源码中定位。
6. 站点构建通过：`npm run build`。

## 8. 进度维护规则

统一进度台账：`docs/ventus-progress.md`。

每次新增/更新文章后必须同步以下信息：

1. `已完成`：新增条目（标题、链接、完成日期、简述）。
2. `进行中`：当前正在写的文章状态。
3. `待办`：根据最新理解调整优先级。
4. `阻塞项`：记录缺失资料、待验证点、环境问题。

状态字段仅使用：`todo` / `doing` / `done` / `blocked`。

## 9. 推荐推进顺序

1. 总览：架构地图与执行链路。
2. CTA 调度系统。
3. Warp 调度与 Pipeline。
4. 内存系统（L1/L2/MMU/LSU）。
5. 仿真框架与测试体系。

可根据学习重点动态调整，但每篇必须更新进度台账。
